# ADR-0001: ファイル一覧画面はローカル/リモート分離ではなく統一ビュー + 同期ステータスで表現する

## 背景と課題 (Context and Problem Statement)

本アプリは端末ストレージのデータをS3に同期する。同一ファイルが「ローカルのみ存在」「リモートのみ存在」「両方に存在し同期済み」「両方に存在するが差分あり」など複数の状態を取りうる。ユーザーがこれをどう見えるべきかが画面構成に直結する。ファイルの所在ごとに画面を分けるべきか、それとも同期状態を属性として扱い1つのリストで見せるべきかを決める必要がある。

## 決定の駆動要因 (Decision Drivers)

* ユーザーは「ファイル」を場所ではなく実体として認識する（同じファイルを2画面で見たくない）
* 認知負荷の最小化（同期状態を確認するためにタブ切り替えが必要だと分かりにくい）
* 画面数・実装ボリュームの最小化
* 同期状態のバリエーションを破綻なく表現できること
* 将来的にフィルタ・検索・並び替えを入れやすい構造であること

## 検討した選択肢 (Considered Options)

* **Option 1: ローカル/リモート分離画面**（タブまたはドロワーで切替、各画面が独立したリスト）
* **Option 2: 統一ビュー + 同期ステータス**（1つのリストに全ファイルを表示し、各行に同期状態をアイコンやバッジで示す）
* **Option 3: ローカル中心ビュー + リモート専用領域**（メインはローカル、リモートのみのファイルは別タブまたは折りたたみで表示）

## 決定 (Decision Outcome)

採用：「Option 2: 統一ビュー + 同期ステータス」。

理由：ユーザーが認識する単位は「ファイル」であり、ローカルかリモートかは属性に過ぎない。状態を属性として扱うことで2重表示・タブ切り替え操作・同一ファイルの突き合わせ確認が不要になる。クラウドストレージUI（Google Drive、Dropbox等）の標準的な慣習にも整合する。

### 結果 (Consequences)

* Good: 認知負荷が低い。同じファイルを別画面で探す手間がない
* Good: 画面数が減り、実装・テスト・保守コストが下がる
* Good: フィルタ機能（"未同期のみ"、"競合のみ"）が後付けで自然に追加できる
* Bad: 同期ステータスの状態設計（同期済み / アップロード待ち / ダウンロード待ち / 競合 / エラー など）を最初に固める必要がある
* Bad: ステータス表現（アイコン・色・バッジ）に十分な視認性がないと、状態が伝わらず混乱の元になる

### 確認方法 (Confirmation)

* ファイル一覧画面が1つだけ存在し、ローカル専用画面・リモート専用画面は存在しない（コードレビューで確認）
* リスト各行に同期ステータスを表す視覚要素（アイコン/バッジ/ラベル）が必ず含まれる（ウィジェットテストで検証）
* ステータスの種類が enum で定義され、UIがその全パターンに対応している（ユニットテスト + ウィジェットテスト）

## 各選択肢の利点・欠点 (Pros and Cons of the Options)

### Option 1: ローカル/リモート分離画面

* Good: 各画面の責務が明確で、データソース単位の実装が直感的
* Good: ローカル/リモートで操作（削除、共有等）が大きく異なる場合に混乱しにくい
* Bad: 同じファイルが2画面に現れ、ユーザーが「これは同じファイルか？」を毎回判定する負荷
* Bad: 同期状態の確認に画面遷移が必要
* Bad: 画面数が増え、状態管理・ナビゲーション・テストの複雑度が上がる

### Option 2: 統一ビュー + 同期ステータス

* Good: ユーザーの認識（ファイル = 実体）と画面構造が一致する
* Good: 主要なクラウドストレージUIと同じ慣習で学習コストが低い
* Good: 単一のリストに対するフィルタ・検索・並び替えが自然に実装できる
* Neutral: ステータスの状態空間設計が必須（後から追加するより最初に整理する方が結果的にコスト低）
* Bad: ステータス表現のデザインを誤ると状態が伝わらない

### Option 3: ローカル中心ビュー + リモート専用領域

* Good: ローカル中心のユースケース（端末上で完結する操作が中心）には素直
* Bad: リモートのみのファイルが目立たない場所に追いやられ、見つけづらい
* Bad: Option 1 と Option 2 の中間で、両者のデメリットを部分的に引き継ぐ
* Bad: 「どちらの領域に出るか」の判定ルールが追加で必要

## 補足情報 (More Information)

* 同期ステータスの具体的な状態定義は別ADRで詳細化する想定（例：競合解決ポリシー、エラー再試行ポリシー）
* 参考UI: Google Drive、Dropbox、iCloud Drive のいずれもファイル単位での統一ビュー + ステータス表現を採用している
