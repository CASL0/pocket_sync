# ADR Creator (MADR v4 形式)

アーキテクチャ上の決定を、その場の会話・PRコメント・誰かの記憶ではなく、コードの隣に残るドキュメントとして記録するためのスキル。MADR v4 (Markdown Architectural Decision Records, 2024-09 リリース) を採用し、本文は日本語で書く。

## なぜMADR v4 なのか

Nygardの原型ADRは軽量だが、AI支援開発では「なぜその選択肢が候補に挙がったのか」「他の案と比べてどう優れていたのか」が後から問われることが多い。MADR v4 は `Decision Drivers` と `Pros and Cons of the Options` を明示的に持つため、判断の根拠を再現しやすく、将来の自分やチームメンバー（人間・AI問わず）が文脈を取り戻しやすい。

v4 では必須セクションが **Title / Context and Problem Statement / Considered Options / Decision Outcome** の4つに絞られ、それ以外（`Decision Drivers`、`Consequences`、`Confirmation`、`Pros and Cons of the Options`、`More Information`）はすべて任意。決定の重さに応じて取捨選択する。

## いつ起動するか

**明示的なシグナル**（ユーザーが直接ADRを求めている）
- 「ADRに残して」「ADRを書いて」「これをADR化したい」
- 「アーキテクチャ決定として記録したい」
- 「なぜXを選んだか後で見返せるようにしたい」

**暗黙的なシグナル**（決定の瞬間を検知 — 自動で書かず、ADR化を提案する）
- 複数のフレームワーク・ライブラリ・サービスを比較し結論に至った（例：「Riverpodで行く」「S3 + Cognito Identity Poolで行く」）
- データモデリング・スキーマ設計の選択（正規化／非正規化、RDB／NoSQL）
- 認証・認可方式の確定
- アーキテクチャパターンの選定（MVVM、Clean Architecture、レイヤ分割）
- インフラ・デプロイ構成の確定
- API設計の方針確定（REST／GraphQL、バージョニング、エラーレスポンス）

**参照のシグナル**（既存ADRを読む）
- 「なぜX を選んだの？」「Xを採用した経緯は？」
- 「以前どういう決定をしたか確認したい」

暗黙的シグナルの場合は、勝手にファイルを作らず「これはADRに残しておきますか？」と一度確認する。決定が小さい（変数名、フォーマット選択など）と感じたら ADR化を提案しない — ADRはノイズが増えると価値が下がる。

## ディレクトリ構成

初回起動時、`docs/adr/` が存在しなければユーザーに確認の上で以下を作成する。

```
docs/
└── adr/
    ├── README.md           ← ADRインデックス（テーブル形式）
    ├── template.md         ← MADRテンプレートのコピー（手動利用用）
    ├── 0001-xxx.md
    ├── 0002-yyy.md
    └── ...
```

`template.md` の中身は `assets/template.md` をコピーする。

## テンプレートの選択

MADR v4 は3種類の公式テンプレートを持つ。決定の重さで使い分ける。

| バリアント | 含まれるセクション | 想定用途 |
|---|---|---|
| **full** | 全セクション（必須4 + 任意5） | 採用フレームワーク・認証方式・インフラ選定など、長期的に影響が残る重要な決定 |
| **minimal** | 必須4 + Consequences | 中規模の決定。トレードオフは記録したいが、駆動要因や各選択肢の詳細比較までは要らない場合 |
| **bare** | 必須4のみ | 軽い決定。結論と選択肢が分かれば十分 |

迷ったら full を使い、不要なセクションを後で削るほうが安全。途中で「このADRは詳細まで書く価値がない」と判明したら minimal や bare に格下げする。

## MADRテンプレート（full）

`assets/template.md` を参照すること。骨子は以下：

```markdown
---
status: proposed
date: YYYY-MM-DD
decision-makers: [名前または役割]
consulted: [意見を聞いた相手 — 任意]
informed: [周知した相手 — 任意]
---

# ADR-NNNN: 短く問題と解決を表すタイトル

## 背景と課題 (Context and Problem Statement)

何が起きているのか、何を決める必要があるのか、2〜3文または短いストーリー形式で。

## 決定の駆動要因 (Decision Drivers)

* {要因1：制約・要求・品質特性など}
* {要因2}
* ...

## 検討した選択肢 (Considered Options)

* {選択肢1のタイトル}
* {選択肢2のタイトル}
* {選択肢3のタイトル}

## 決定 (Decision Outcome)

採用：「{選択肢1のタイトル}」。理由は {判断の根拠を1〜3文で}。

### 結果 (Consequences)

* Good: {ポジティブな帰結}
* Bad: {ネガティブな帰結・トレードオフ}

### 確認方法 (Confirmation)

この決定が守られていること／実装されていることをどう確認するか（例：CI上のチェック、レビュー観点、E2Eテスト）。

## 各選択肢の利点・欠点 (Pros and Cons of the Options)

### {選択肢1のタイトル}
* Good: {根拠 a}
* Good: {根拠 b}
* Neutral: {根拠 c}
* Bad: {根拠 d}

### {選択肢2のタイトル}
* Good: ...
* Bad: ...

## 補足情報 (More Information)

参考リンク、関連Issue/PR、再考のトリガー条件、検証データなど（任意）。
```

## MADRテンプレート（minimal）

```markdown
---
status: accepted
date: YYYY-MM-DD
decision-makers: [名前]
---

# ADR-NNNN: タイトル

## 背景と課題 (Context and Problem Statement)
## 検討した選択肢 (Considered Options)
## 決定 (Decision Outcome)

採用：「X」。理由：...

### 結果 (Consequences)
* Good: ...
* Bad: ...
```

## MADRテンプレート（bare）

```markdown
# ADR-NNNN: タイトル

## 背景と課題 (Context and Problem Statement)
## 検討した選択肢 (Considered Options)
* X
* Y

## 決定 (Decision Outcome)

採用：「X」。理由：...
```

どのバリアントにするか迷ったら、ユーザーに「この決定は将来詳細を追跡したい？それとも結論だけで十分？」と聞く。

## ワークフロー

### 新規ADRを書く

1. **初期化**（初回のみ）: `docs/adr/` が無ければユーザーに確認の上、ディレクトリ・`README.md`（インデックス）・`template.md` を作成する。承認が無ければ作らない。
2. **決定の特定**: 何を決めようとしているのか核を抽出する。タイトルは「Xを採用する」「XよりYを優先する」など、解決と問題の両方が分かる短文にする。
3. **背景の収集**: 何がきっかけでこの決定が必要になったのか、どんな制約があるのかをユーザーから引き出す。会話履歴から拾える場合は明示確認だけで足りる。
4. **駆動要因の整理**: なぜこの決定が必要なのか（性能・セキュリティ・運用コスト・チームスキル等）を箇条書きで。
5. **選択肢の列挙**: 検討した／棄却した案を網羅する。「他に検討したものは？」と必ず問う — 1案しか挙げないADRは価値が薄い。
6. **採用案と根拠**: 駆動要因と各選択肢の照合から、採用案を1〜3文で言い切る。
7. **結果とトレードオフ**: ポジティブ・ネガティブ両方を正直に書く。トレードオフが無いと書いてあるADRは信用されない。
8. **番号付け**: `docs/adr/` をスキャンして既存の最大番号 + 1 を採番。ファイル名は `NNNN-kebab-case-title.md`（タイトル部分は英語のkebab-caseまたはローマ字）。
9. **ドラフト提示と承認**: ファイルを書く前に内容をユーザーに見せて承認を得る。承認なしに書き込まない。修正要求があれば反映してから書く。
10. **インデックス更新**: `docs/adr/README.md` のテーブルに行を追加する。
11. **コミット提案**（任意）: このプロジェクトはConventional Commits + 絵文字を使っているため、コミットを提案する場合は `docs(adr): :memo: ADR-NNNN <タイトル>` の形式を提示する。

### 既存ADRを参照する

1. `docs/adr/` が無ければ「ADRはまだありません。記録を始めますか？」と返す。
2. あれば `docs/adr/README.md` のインデックスをスキャンし、関連するエントリを特定する。
3. 該当ADRの「背景と課題」「決定」を中心に提示する。全文が必要そうならユーザーに確認してから読み込む。
4. 該当が見つからなければ「該当するADRは見つかりませんでした。今から記録しますか？」と返す。

## インデックス形式 (README.md)

```markdown
# Architecture Decision Records

このプロジェクトのアーキテクチャ決定を MADR 形式で記録する。テンプレートは [template.md](template.md) を参照。

| ADR | タイトル | ステータス | 日付 |
|-----|---------|-----------|------|
| [0001](0001-use-riverpod.md) | 状態管理にRiverpodを採用する | accepted | 2026-04-30 |
| [0002](0002-s3-with-cognito.md) | 同期先にS3 + Cognito Identity Poolを採用する | proposed | 2026-05-02 |
```

## ステータス遷移

```
proposed → accepted → [deprecated | superseded by ADR-NNNN]
                  ↓
                rejected
```

- **proposed**: 議論中。チームの合意が未確定。
- **accepted**: 採用済み。実装・運用に反映されている／されつつある。
- **rejected**: 検討の結果採用しなかった案を残す場合に使う。
- **deprecated**: もはや関係なくなった（機能削除など）。
- **superseded by ADR-NNNN**: より新しいADRに置き換えられた。必ず置き換え先を明記する。

ステータスが `superseded` または `deprecated` になった旧ADRは削除しない — 履歴として残す。インデックスのステータス列だけ更新する。

## 良いADRの条件

**やる**
- 具体的に書く（「ORMを使う」ではなく「Driftを採用する」）
- 「なぜ」を残す — 「何を」よりも判断の根拠が後から効いてくる
- 棄却した選択肢を必ず書く — 「これしか選択肢がなかった」と書いてあっても通常は嘘
- トレードオフを正直に書く — どんな決定にも対価がある
- 短く保つ — 全体を2〜3分で読み切れる量に
- 現在形で書く — 「Xを採用する」（「Xを採用しようと思う」ではない）

**やらない**
- 自明な決定を記録しない（変数命名、コードフォーマッタの選択など）
- 長文エッセイにしない — 背景セクションが10行を超えるなら整理する
- 選択肢を1つしか書かない — 比較の痕跡が無いADRは判断根拠の信頼性が低い
- 過去の決定を遡って書くときは、元の決定日を明記する（バックフィルである旨も）
- 古くなったADRを放置しない — 置き換えられたADRは `superseded by` を必ず付ける

## 記録すべき決定のカテゴリ（参考）

| カテゴリ | 例 |
|---------|---|
| 技術選定 | フレームワーク、言語、データベース、クラウドプロバイダ |
| アーキテクチャ | MVVM/Clean Architecture、モノリス／マイクロサービス、レイヤ構成 |
| 状態管理 | Riverpod / BLoC / Provider / Redux |
| API設計 | REST / GraphQL、バージョニング、エラーフォーマット |
| データ | スキーマ設計、正規化判断、キャッシュ戦略、ローカルDB選定 |
| 認証・認可 | OAuth2 / Cognito / Firebase Auth、トークン保管 |
| インフラ | デプロイ先、CI/CD、監視、ログ集約 |
| セキュリティ | 暗号化方式、シークレット管理、MASVS対応方針 |
| テスト | テストフレームワーク、カバレッジ目標、E2E方針 |
